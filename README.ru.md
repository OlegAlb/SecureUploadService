# Secure Upload Service (Сервис безопасной загрузки)

Ориентированный на продакшен микросервис безопасной загрузки файлов с защитой от произвольной загрузки файлов (Arbitrary File Upload) и асинхронным аудитом безопасности.

---

# Возможности

## Безопасная потоковая загрузка

- Настоящий конвейер загрузки с приоритетом потоковой передачи 
- Файлы не загружаются полностью в оперативную память (RAM)
- Инкрементальное хеширование по алгоритму SHA-256
- Ограничение размера файла прямо во время потоковой обработки 

---

## MIME-валидация через Magic Bytes (сигнатуры файлов)

Сервис не доверяет:

- расширению файла;
- заголовку `Content-Type`, переданному клиентом.

Настоящий MIME-тип определяется через анализ сигнатур.

Пример:

```text
malware.php.jpg
```

будет определён как:

```text
application/x-php
```

и заблокирован.

---

## Защита от удаленного выполнения кода (RCE)

Загружаемые файлы защищены с помощью:

- случайных 64-байтовых имен файлов;
- хранения вне директорий с возможностью выполнения;
- прав доступа на уровне файловой системы (`0600`);
- отсутствия путей, контролируемых пользователем.

---

## Асинхронный аудит безопасности

Каждая загрузка (успешная или заблокированная) проходит следующий путь:

```text
API
  |
  v
Redis Queue
  |
  v
Audit Worker
  |
  v
ClickHouse
```

Журналы аудита записываются в ClickHouse пакетами для эффективного хранения и аналитики.

---

## Стратегия пакетной вставки 

Воркер накапливает события аудита в памяти и периодически выполняет пакетную вставку в ClickHouse.

Это позволяет избежать ситуации:

```text
1 upload = 1 INSERT
```

что неэффективно для аналитических СУБД.

---

## Распределенное ограничение частоты запросов 

Ограничение частоты запросов на базе Redis:

```text
100 uploads / minute / IP
```

защищает сервис от спама загрузками и базовых DoS-атак.

---

## Эндпоинт проверки работоспособности

```http
GET /health
```

Проверяет:

- Соединение с Redis
- Соединение с ClickHouse

# Конвейер загрузки

```text
Incoming Stream
       |
       +--> first 4KB buffer
       |
       +--> magic bytes validation
       |
       +--> SHA-256 hashing
       |
       v
Secure Write Stream
```

В памяти буферизуются только первые байты, необходимые для MIME-валидации. Остальная полезная нагрузка стримится напрямую на диск.

# Решения по обеспечению безопасности

## Почему magic bytes вместо расширений файлов?

Расширение файла полностью контролируется пользователем, и его легко подделать

Пример:

```text
shell.php.jpg
```

Валидация по magic bytes анализирует реальную бинарную сигнатуру файла.

---

## Почему используются случайные имена файлов?

Предсказуемые имена файлов могут сделать возможными:

- прямой поиск и обнаружение файлов;
- попытки вредоносного запуска;
- атаки методом перебора.

Поэтому файлы сохраняются со случайными криптографическими идентификаторами.

---

## Почему audit logging асинхронный?

Журналирование аудита не должно блокировать конвейер загрузки.

API публикует событие в Redis и сразу возвращает ответ .

Тяжелые аналитические записи делегируются отдельному процессу-воркеру.

---

## Почему ClickHouse?

ClickHouse оптимизирован для:

- нагрузок с преобладанием добавления данных;
- аналитических запросов;
- вставок с высокой пропускной способностью;
- хранения логов и событий.

Это делает его отличным выбором для конвейеров аудита.

---

# Модель угроз

Сервис спроектирован для предотвращения и снижения рисков (mitigation) следующих угроз:

- Произвольная загрузка файлов (Arbitrary File Upload)
- Подмена MIME-типа (MIME spoofing)
- Выход за пределы директории (Path Traversal)
- Базовые DoS-атаки на загрузку файлов
- Удаленное выполнение кода (RCE) через предсказуемые имена файлов
- Синхронные «узкие места» при аудите (Synchronous audit bottlenecks)
